Multi-Landlord Isolation Audit

Entity Who can read Isolation mechanism Status
properties landlord all queries filter WHERE landlord_id = req.user.id ✓ Isolated
rooms landlord each operation first finds Property WHERE landlord_id = req.user.id — room inherits ownership via its property ✓ Isolated
tenants (list/edit/invite) landlord all queries filter WHERE landlord_id = req.user.id ✓ Isolated
tenants (own profile) tenant WHERE user_id = req.user.id AND status = 'active' ORDER BY id DESC — picks the latest active record, ignores vacated history ✓ Isolated
payments (landlord view) landlord all queries filter WHERE landlord_id = req.user.id ✓ Isolated
payments (tenant view) tenant tenant can only request their own tenant_id, verified via tenant.user_id === req.user.id ✓ Isolated
complaints (landlord view) landlord all queries filter WHERE landlord_id = req.user.id ✓ Isolated
complaints (tenant view) tenant verifies tenant.user_id = req.user.id before returning complaints for that tenant_id ✓ Isolated
receipts landlordtenant landlord: payment.landlord_id = req.user.id · tenant: active tenant lookup prevents vacated-record shadow ✓ Isolated
reports/summary landlord all counts/sums filtered by landlord_id = req.user.id ✓ Isolated
reports/admin-summary admin only roleCheck(['admin']) gate — shows platform totals, no per-landlord data exposed ✓ Isolated

Key rule enforced throughout: Each landlord's data is an independent island. A landlord's properties, rooms, tenants, payments, complaints, and reports are all filtered by their own user.id. No landlord can read, write, or affect another landlord's records. Tenants only see their own profile, their own payments, and their own complaints — they cannot browse other tenants.
